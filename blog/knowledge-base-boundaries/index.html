企业知识库落地边界：RAG 不是把文档扔进去

很多企业知识库项目失败，不是因为向量数据库不好，也不是因为模型能力不够，而是因为文档本身没有业务边界。

如果 SOP 过期、命名混乱、适用条件不清、答案没人负责更新，把这些文档全部丢进 RAG，只会得到一个更会包装混乱的问答机器人。

知识库首先是责任系统

一个可落地的知识库，至少要回答：

  • 这份知识适用于什么场景。
  • 谁负责维护。
  • 最后一次验证是什么时候。
  • 遇到冲突以哪份为准。
  • 哪些问题不能自动回答，必须升级给人。

没有这些字段，问答准确率很难长期稳定。

文档切片不是越碎越好

很多人以为 RAG 的关键是把文档切得足够细。切片当然重要，但更重要的是切片语义是否完整。

一个好的 SOP 切片应该包含：

  • 问题现象。
  • 适用条件。
  • 排查步骤。
  • 风险提示。
  • 升级条件。
  • 相关系统或负责人。

如果只切出一段孤立文字，模型可能检索到了内容，却不知道它适用于哪个场景。

问答不是唯一输出

企业知识库不一定要以聊天机器人为中心。很多时候，更有价值的输出是：

  • 工单摘要。
  • 推荐 SOP。
  • 缺失信息清单。
  • 标准回复草稿。
  • 升级路径建议。
  • 复盘模板。

这些输出更容易进入真实工作流，也更容易验收。

适合先做知识库 Copilot 的场景

优先选择这些场景：

  1. 问题类型重复，历史文档或工单足够多。
  2. 一线人员需要快速找到下一步动作。
  3. 答案允许人工确认，而不是必须完全自动。
  4. 文档可以持续更新。
  5. 有明确升级路径。

不适合的场景包括：文档大量过期、责任人缺失、问题高度个性化、答案涉及强合规风险但没有审核流程。

最小可行版本

一个知识库 Copilot 的第一版可以很小：

输入一条工单 -> 提取关键信息 -> 检索相关 SOP -> 输出排查路径和回复草稿 -> 人工确认

第一版不需要覆盖所有问题。它只需要证明：在一个高频问题类型上，能减少查找时间、减少重复解释，并让新人更快找到下一步动作。

真正的交付物

知识库项目的交付物不应该只有一个问答入口，还应该包括：

  • 文档字段规范。
  • SOP 切片规则。
  • 不回答清单。
  • 升级规则。
  • 反馈入口。
  • 更新节奏。
  • 典型问题测试集。

RAG 是技术实现，知识库是组织流程。两者必须一起设计，项目才有机会长期可用。
